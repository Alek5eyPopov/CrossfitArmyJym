- **Файлы:** `ScheduleAdapter.java`, `HomeViewModel.java`, `ScheduleViewModel.java`, `WodRepository.java`
- **Описание:** `SimpleDateFormat` — не потокобезопасный класс, используется в статических/общих методах. Рекомендуется `FastDateFormat` из Apache Commons или создавать новый экземпляр при каждом вызове.

### 🔴 Критические проблемы производительности

#### 1. Room запросы на главном потоке
`SQLiteException` в debug-сборках:

| Файл | Строка | Вызов |
|------|--------|-------|
| `ClassRepository.java` | 62 | `dao.getClassesFromDate(date)` |
| `ClassRepository.java` | 75 | `dao.getClassesByTrainer(trainerId)` |
| `ClassRepository.java` | 130 | `dao.getClassesFromDate(date)` (fallback) |
| `BookingRepository.java` | 81 | `bookingDao.getBookingsByUser(userId)` |
| `BookingRepository.java` | 176, 188 | `bookingDao.getBookingsByUser(userId)` (fallback) |
| `WodRepository.java` | 73 | `dao.getWodByDate(today)` |
| `WodRepository.java` | 90 | `dao.getWodById(wodId)` |
| `WodRepository.java` | 127-129 | `dao.getAllWods()` |

**Решение:** Использовать LiveData/Flow из Room или обернуть в AsyncTask/Coroutine:
```java
// Вариант 1: LiveData из DAO
@Query("SELECT * FROM cached_classes ORDER BY scheduled_start ASC")
LiveData<List<GymClassEntity>> getAllClasses();

// Вариант 2: AsyncTask
new AsyncTask<Void, Void, List<GymClassEntity>>() {
    @Override protected List<GymClassEntity> doInBackground(Void... voids) {
        return dao.getClassesFromDate(date);
    }
    @Override protected void onPostExecute(List<GymClassEntity> result) {
        // обработка результата
    }
}.execute();
```

#### 2. `notifyDataSetChanged()` вместо точечного обновления
- **Файл:** `ScheduleAdapter.java`
- **Строка:** 96
- **Код:** `notifyDataSetChanged()`
- **Проблема:** При любом изменении списка перерисовываются ВСЕ элементы.
- **Решение:** Использовать `ListAdapter` с `DiffUtil.ItemCallback`:
```java
public class ScheduleAdapter extends ListAdapter<GymClass, ScheduleAdapter.ViewHolder> {
    private static final DiffUtil.ItemCallback<GymClass> DIFF_CALLBACK = ...;
    public ScheduleAdapter() { super(DIFF_CALLBACK); }
}
```

### 🟡 Проблемы средней серьезности

#### 3. Утечки памяти через анонимные Callback
- **Файлы:** `ClassRepository.java`, `BookingRepository.java`, `WodRepository.java`
- **Всего:** 6 анонимных `Callback` + 6 `Runnable` + 3 лямбды
- **Проблема:** Анонимные классы содержат неявную ссылку на внешний класс (Activity/Application). Если Activity уничтожена, callback продолжает висеть.
- **Решение:** Использовать статические вложенные классы или WeakReference, или отменять запросы в `onCleared()`.

#### 4. Отсутствие отмены запросов при уничтожении ViewModel
- **Файлы:** Все ViewModel (8 шт) — `HomeViewModel`, `ScheduleViewModel`, `BookingsViewModel`, `TrainerClassesViewModel`, `AttendanceViewModel`, `WodEditorViewModel`, `AdminUsersViewModel`, `AdminScheduleViewModel`, `AdminStatisticsViewModel`, `AdminWodViewModel`
- **Решение:** Добавить `@Override onCleared()` и отменять все выполняющиеся запросы:
```java
@Override
protected void onCleared() {
    super.onCleared();
    // отмена всех выполняющихся Call
}
```

#### 5. OkHttp без кэширования
- **Файлы:** `AttendanceViewModel.java`, `WodEditorViewModel.java`, `AdminUsersViewModel.java`, `AdminScheduleViewModel.java`, `AdminWodViewModel.java`, `AdminStatisticsViewModel.java`
- **Решение:** Добавить `HttpCache`:
```java
File cacheDir = new File(context.getCacheDir(), "http_cache");
Cache cache = new Cache(cacheDir, 10 * 1024 * 1024); // 10MB
httpClient = new OkHttpClient.Builder().cache(cache).build();
```

### 🟢 Низкий приоритет

#### 6. AndroidViewModel без необходимости
- **Файлы:** `AttendanceViewModel.java`, `WodEditorViewModel.java`
- **Описание:** Используют `AndroidViewModel`, но не вызывают `getApplication()`. Требуется только `PreferencesManager`.
- **Решение:** Заменить на `ViewModel` и получать Application через конструктор при необходимости.

#### 7. Нет пагинации в ScheduleFragment
- **Файл:** `ScheduleFragment.java`
- **Описание:** Загружает все занятия сразу без пагинации. При большом количестве данных — зависание UI.
- **Решение:** Использовать `Paging 3` библиотеку или ручную пагинацию через `limit`/`offset` параметры.

#### 2. Паттерн ViewModel (повторяется 8+ раз)
**Затронутые файлы:** `HomeViewModel`, `ScheduleViewModel`, `BookingsViewModel`, `TrainerClassesViewModel`, `AttendanceViewModel`, `WodEditorViewModel`, `AdminUsersViewModel`, `AdminScheduleViewModel`, `AdminStatisticsViewModel`, `AdminWodViewModel`

**Повторяющийся код:**
```java
private static final String TAG = "...";
private final MutableLiveData<Boolean> isLoading = new MutableLiveData<>(false);
private final MutableLiveData<String> errorMessage = new MutableLiveData<>();
```

**Рекомендация:** Выделить базовый класс:
```java
public abstract class BaseViewModel extends AndroidViewModel {
    protected final MutableLiveData<Boolean> isLoading = new MutableLiveData<>(false);
    protected final MutableLiveData<String> errorMessage = new MutableLiveData<>();
    
    public BaseViewModel(@NonNull Application application) { super(application); }
    public LiveData<Boolean> getIsLoading() { return isLoading; }
    public LiveData<String> getErrorMessage() { return errorMessage; }
}
```

#### 3. Паттерн Fragment (повторяется 10+ раз)

**Повторяющийся шаблон:**

```java
@Nullable @Override
```

#### 4. DAO методы (повторяются в 9 DAO)
**Затронутые файлы:** Все 9 DAO

**Повторяющиеся методы:**
- `insert(T entity)` — во всех DAO (формат одинаковый)
- `insertAll(List<T>)` — во всех DAO
- `clearAll()` — во всех DAO
- `clearOldEntries(long timestamp)` — в 4 DAO

**Рекомендация:** Выделить базовый интерфейс DAO:
```java
public interface BaseDao<T> {
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    void insert(T entity);
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    void insertAll(List<T> entities);
    @Query("DELETE FROM table_name")
    void clearAll();
}
```

#### 5. Layout основных активностей (3 файла)
**Файлы:** `activity_client_main.xml`, `activity_trainer_main.xml`, `activity_admin_main.xml`

**Проблема:** 100% идентичная структура (CoordinatorLayout + FragmentContainerView + BottomNavigationView). Отличаются только `id`, `navGraph` и `menu`.

**Рекомендация:** Один общий layout + установка navGraph/menu программно:
```xml
<!-- activity_base_main.xml -->
<androidx.coordinatorlayout.widget.CoordinatorLayout>
    <androidx.fragment.app.FragmentContainerView android:id="@+id/nav_host_fragment" />
    <com.google.android.material.bottomnavigation.BottomNavigationView android:id="@+id/bottom_navigation" />
</androidx.coordinatorlayout.widget.CoordinatorLayout>
```
```java
// В Activity:
NavHostFragment navHost = (NavHostFragment) getSupportFragmentManager()
    .findFragmentById(R.id.nav_host_fragment);
```
