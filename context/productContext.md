# Product Context: CrossFit Gym Manager

## Why This Project Exists
CrossFit залы сталкиваются с хаосом:
- Атлеты не знают технику упражнений (травмы)
- Тренеры тратят часы на создание WOD вручную в Excel
- Нет персонального подхода к группам разного уровня
- Администраторы не могут контролировать качество тренировок

Приложение решает:
- Централизованная база знаний (упражнения с техникой)
- Групповая динамика (соревнование внутри команды)
- Разделение ролей (дисциплина и безопасность)

## Problems It Solves
| Проблема | Решение |
|----------|---------|
| Атлет не знает, как делать упражнение | База упражнений с видео и описанием |
| Тренер тратит 2 часа на создание WOD на неделю | Шаблоны WOD + копирование/быстрое редактирование |
| Нет соревновательного элемента внутри группы | Лидерборд по группе + общий рейтинг зала |
| Админ не может проверить работу тренера | Полный доступ ко всем WOD и результатам |
| Атлет тренируется не по своей программе | WOD привязаны к группе |

## How It Should Work

### Сценарий 1: Атлет готовится к тренировке
1. Открывает приложение → видит WOD своей группы на сегодня
2. Нажимает на упражнение → открывается база: как делать, видео, техника
3. Идёт на тренировку, выполняет WOD

### Сценарий 2: Атлет записывает результат
1. После WOD открывает результат
2. Выбирает тип результата: время, вес, повторения
3. Вводит значение (например: "15:30" для For Time)
4. Приложение проверяет: это новый PR? → поздравление
5. Лидерборд обновляется автоматически

### Сценарий 3: Тренер создаёт WOD
1. Выбирает "Create WOD"
2. Добавляет название: "Hero WOD - Murph"
3. Выбирает формат: For Time
4. Добавляет блоки упражнений (выбор из базы):
    - Run 1 mile
    - 100 Pull-ups
    - 200 Push-ups
    - 300 Squats
    - Run 1 mile
5. Назначает группе: "Morning Crew"
6. Устанавливает дату: завтра
7. Публикует → WOD виден только группе

### Сценарий 4: Администратор управляет группами
1. Создаёт группу "Competition Team"
2. Назначает тренера (из списка пользователей с ролью тренер)
3. Добавляет атлетов (+ поиск)
4. Меняет роль пользователя: атлет → тренер
5. Блокирует неактивного пользователя

## User Experience Goals
- **Атлет**: понятные карточки WOD, быстрый ввод результата (< 3 кликов)
- **Тренер**: визуальный конструктор WOD (drag & drop упражнений)
- **Админ**: дашборд с активностью зала (графики посещаемости)
- **Все роли**: оффлайн режим + синхронизация при подключении

## Общие требования
- Весь код на Java.
- Пошаговый план выполнения находится в файле activeContext.md. Выполнение каждого шага по команде человека-разработчика
- Использовать MVVM: Activity/Fragment – View – ViewModel – Repository – Supabase API.
- Для всех сетевых запросов использовать Retrofit с Gson.
- Для аутентификации – Supabase Auth (email/пароль).
- Для офлайн-режима – Room (кэш основных таблиц).
- Для навигации – Navigation Component + BottomNavigationView (для клиента), отдельные Activity для разных ролей.
- Для работы с датами – использовать `java.time` (API 26+) или ThreeTenABP.
- Все строки, цвета, размеры выносить в ресурсы.

## Правила безопасности
- Никогда не хранить `anon key` или `service role key` в коде – использовать `local.properties` и BuildConfig.
- JWT-токен хранить в `EncryptedSharedPreferences`.
- Все запросы к Supabase отправлять с заголовком `apikey` и `Authorization: Bearer <token>`.

## Правила для ИИ-агента
- Предлагать полные реализации классов с комментариями.
- Не использовать устаревшие API.
- Для каждого нового функционального шага предлагать сначала изменения в схеме БД (SQL), затем код Android.
- Предупреждать о необходимости обновления RLS-политик в Supabase.
- Все API-вызовы должны быть асинхронными (enqueue или Callback).
- При реализации ролей проверять права на стороне клиента и ожидать их проверки на сервере (RLS).

## Логирование
Пожалуйста, веди подробный лог всех своих действий в следующем формате:

[УРОВЕНЬ] ВРЕМЯ | ДЕЙСТВИЕ | ПОДРОБНОСТИ | РЕЗУЛЬТАТ | ДЛИТЕЛЬНОСТЬ

Где УРОВЕНЬ: INFO, DEBUG, WARN, ERROR
ВРЕМЯ: ЧЧ:ММ:СС.ммм
ДЕЙСТВИЕ: краткое название (API_CALL, DB_QUERY, FILE_READ, PARSE, THINK)
ПОДРОБНОСТИ: параметры, URL, запрос
РЕЗУЛЬТАТ: статус (OK/FAIL) и краткий итог

Пример:
[INFO] 10:30:00.123 | TASK_START | обновление данных заказа | orderId=12345 | OK
[DEBUG] 10:30:00.456 | DB_QUERY | SELECT * FROM orders WHERE id=12345 | найдено 1 запись | 234ms
[INFO] 10:30:01.789 | TASK_END | обновление заказа | успешно, обновлено 3 поля | 1.6s

логгирование осуществлять в файле progress.md
