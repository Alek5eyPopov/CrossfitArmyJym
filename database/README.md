# Supabase database

Настройка фирменного письма и Android-перехода после подтверждения email описана в
[`EMAIL_CONFIRMATION_SETUP.md`](EMAIL_CONFIRMATION_SETUP.md).

## Порядок применения

Для нового проекта:

1. Выполнить `supabase_schema.sql` в Supabase SQL Editor.
2. Выполнить файлы из `migrations` в порядке их имён.
3. Выполнить проверки из раздела ниже под тестовыми пользователями.

Для существующего проекта выполняются только ещё не применённые миграции.
Миграции нельзя запускать с клиентским `anon key`: их выполняет владелец проекта
через SQL Editor или доверенный CI с административным доступом.

## Тестовые данные

`seed_demo.sql` заполняет development/staging-проект демонстрационными данными:

- двумя группами;
- шестью упражнениями;
- тремя WOD относительно текущей даты;
- прошедшими и будущими занятиями;
- записями, посещаемостью и результатами существующих атлетов.

Перед запуском:

1. Зарегистрировать хотя бы одного атлета и одного тренера через Supabase Auth.
2. Для тренера установить `profiles.role = 'trainer'`.
3. При необходимости указать их email в начале `seed_demo.sql` вместо `NULL`.
4. Выполнить весь `seed_demo.sql` в Supabase SQL Editor.
5. Перезапустить приложение или нажать кнопки обновления на экранах.

Скрипт предназначен только для тестового проекта. Он не создаёт `auth.users`,
не хранит пароли и при повторном запуске заменяет только записи со своими
фиксированными demo-ID. Все активные профили с ролью `athlete` распределяются
по двум демонстрационным группам.

## Этап 2

`migrations/20260611_stage2_security.sql`:

- запрещает назначать роль через signup metadata;
- блокирует изменение `role`, `group_id`, `is_active` и email самим
  пользователем;
- заменяет рекурсивные проверки `profiles` на security-definer helpers;
- ограничивает политики ролью `authenticated`;
- закрывает прямую запись клиента в `bookings`;
- добавляет атомарные RPC `book_class(UUID)` и `cancel_booking(UUID)`;
- синхронизирует `classes.current_bookings` триггером.

Миграция транзакционная и повторно применимая.

## Этапы 6 и 7

`migrations/20260611_stage6_wod_results.sql` добавляет атомарное создание WOD с
упражнениями, сохранение результата с расчётом PR и групповой лидерборд.

`migrations/20260611_stage7_admin_management.sql` разрешает администратору и
назначенному тренеру читать отключённые группы для последующего редактирования
или повторной активации. Обычные пользователи продолжают видеть только активные
группы.

После применения миграции этапа 7:

1. Администратор должен видеть активные и отключённые группы.
2. Назначенный тренер должен видеть свою отключённую группу.
3. Атлет не должен видеть отключённую группу.

## Ручная проверка

После применения миграции:

1. Зарегистрировать нового пользователя, передав metadata `role=admin`.
   В `profiles.role` всё равно должно сохраниться `athlete`.
2. Под JWT атлета попробовать изменить свой `role`, `group_id` или
   `is_active`. Запрос должен завершиться ошибкой `42501`.
3. Под JWT атлета вызвать:

   ```http
   POST /rest/v1/rpc/book_class
   {"p_class_id":"<class-uuid>"}
   ```

   Должна появиться подтверждённая запись, а `current_bookings` увеличиться на 1.
4. Повторный вызов `book_class` для той же пары пользователь/занятие не должен
   создавать дубликат или увеличивать счётчик.
5. При заполненном занятии следующий вызов должен вернуть
   `Class capacity reached`.
6. Вызвать:

   ```http
   POST /rest/v1/rpc/cancel_booking
   {"p_booking_id":"<booking-uuid>"}
   ```

   Статус должен стать `cancelled`, а счётчик уменьшиться на 1.
7. Проверить, что тренер видит записи и посещаемость только своих занятий, а
   администратор видит все записи.

## Локальная синтаксическая проверка

Вспомогательный скрипт `validate_sql.py` использует `pglast`:

```powershell
python -m pip install -r database/requirements-dev.txt
python database/validate_sql.py
```

## Этап 8: WOD 2.0 и личные PR

`migrations/20260616_stage8_wod_tasks_prs.sql` добавляет основу для новой логики:

- `load_types` — справочник типов нагрузки: AMRAP, EMOM, For Time, Complex, Strength, Tabata, Other;
- `training_tasks` — переиспользуемые задания на базе упражнения, типа нагрузки и текстового описания нагрузки;
- `wod_tasks` — состав конкретного WOD с порядком пунктов, RX-вариантом и Optional-вариантом;
- `personal_records` — история личных рекордов атлета по упражнениям;
- `personal_record_bests` — view с текущим лучшим результатом по каждому упражнению;
- RPC `create_wod_with_tasks(...)` — атомарное создание WOD с несколькими заданиями;
- RPC `submit_personal_record(...)` — добавление нового личного PR без редактирования старой истории.

После применения миграции атлеты могут только добавлять новые PR и WOD-результаты. Политика прямого обновления старых строк `results` удаляется, поэтому исправления старых результатов остаются административной задачей, а история атлета не переписывается с клиента.
